ENHANCEMENT #1 — Replace QLDB with Hash-Chained DynamoDB + S3 Object Lock
--------------------------------------------------------------------------------
ORIGINAL SPEC   : Amazon QLDB as primary compliance ledger
PROBLEM         : QLDB reached end-of-support July 31, 2025. Deprecated service.
                  Cannot build a new production system on a dead AWS service,
                  especially one requiring 7-year compliance retention.

RECOMMENDATION  :
  Every chat event written to DynamoDB gets:
    - A SHA-256 hash of its own content
    - The hash of the previous record chained in (tamper-evidence)
    - An immutable timestamp

  Record N  : { content, timestamp, hash: SHA256(content + prev_hash), prev_hash }
  Record N+1: { content, timestamp, hash: SHA256(content + record_N_hash), prev_hash: record_N_hash }

  Tamper-evidence: if anyone edits record N, its hash breaks,
  which breaks every subsequent record — detectable instantly.

  S3 Object Lock (Compliance Mode) handles 7-year WORM archival.
  This already satisfies SEC Rule 17a-4 and FINRA requirements.
  No QLDB needed.

BENEFIT         : Removes a deprecated service. Eliminates future migration risk.
                  S3 Object Lock is the legally sufficient WORM mechanism.
IMPLEMENTATION  : Medium effort — hash chaining logic in application layer.

ENHANCEMENT #2 — Cold Start Strategy for Lambda (Provisioned Concurrency)
--------------------------------------------------------------------------------
ORIGINAL SPEC   : AWS Lambda — no cold start strategy mentioned
PROBLEM         : Lambda cold starts (1–4 seconds) create visible silence when
                  a user first opens the chat widget. Feels broken. Users leave.

                  Cold start timeline:
                    AWS spins up container → loads Python runtime → imports
                    libraries → THEN runs your code → adds 1–4 seconds BEFORE
                    the LLM even starts responding.

RECOMMENDATION  :
  Dev / Staging:
    → EventBridge rule pings Lambda every 5 minutes (free)
    → Prevents container from being killed during testing

  Production:
    → Provisioned Concurrency = 1 always-warm instance
    → Cost: ~$40/month (EBAM pays AWS bill — negligible vs UX impact)
    → Zero cold starts for that instance

  UI Layer (always, regardless of above):
    → Show "Connecting..." spinner immediately when widget opens
    → Masks any cold start that slips through
    → Users accept a loading state; they do not accept silence

BENEFIT         : Eliminates the most noticeable UX failure point in the product.
IMPLEMENTATION  : Low effort — one config change + UI spinner.


--------------------------------------------------------------------------------
ENHANCEMENT #3 — Drop AppSync, Use API Gateway Only
--------------------------------------------------------------------------------
ORIGINAL SPEC   : API Gateway + AppSync (both specified)
PROBLEM         : AppSync is a GraphQL service. Running both API Gateway (REST)
                  and AppSync (GraphQL) means:
                    - Two separate authentication configurations
                    - Two separate deployment pipelines
                    - Two sets of IAM policies
                    - AppSync charges per query — unnecessary cost
                    - Steep learning curve with no benefit for this use case

RECOMMENDATION  :
  API Gateway HTTP API   → All REST endpoints (chat, admin, leads)
  API Gateway WebSocket  → Real-time streaming if needed in future

  Single service. Single auth. Single deployment pipeline.
  AppSync only makes sense for full GraphQL architecture — not the case here.

BENEFIT         : Simpler infrastructure, lower cost, faster deployment.
IMPLEMENTATION  : Low effort — remove AppSync from architecture, API Gateway only.


--------------------------------------------------------------------------------
ENHANCEMENT #4 — Add Minimal Admin Auth in Phase 1 (Don't Wait for Cognito)
--------------------------------------------------------------------------------
ORIGINAL SPEC   : Authentication deferred entirely to Phase 2 (Cognito)
PROBLEM         : Admin dashboard goes live with ZERO authentication.
                  Anyone with the URL can:
                    - View all captured leads (names, emails, phones)
                    - Edit conversation flows
                    - Wipe data
                  This is a serious security and compliance risk.

RECOMMENDATION  :
  Phase 1 (1 hour to implement):
    → Single API key stored in AWS Secrets Manager
    → All /api/admin/* routes require Authorization header
    → Header: Authorization: Bearer ebam-admin-key-xxxx
    → Key rotatable without code changes

  Phase 2:
    → Replace with AWS Cognito for proper multi-user login
    → Same endpoints, just swap the auth middleware

BENEFIT         : Protects live lead data and flow configs without Cognito complexity.
IMPLEMENTATION  : Low effort — one middleware function + Secrets Manager entry.


--------------------------------------------------------------------------------
ENHANCEMENT #5 — Use Node.js for Lambda Instead of Python
--------------------------------------------------------------------------------
ORIGINAL SPEC   : Lambda (Python/Node.js) — both listed, no preference stated
PROBLEM         : Python cold starts are significantly slower than Node.js.

                  Python Lambda cold start :  1,500 – 4,000 ms
                  Node.js Lambda cold start:    200 –   800 ms

                  Node.js is 3–5x faster to boot. For a chatbot, every
                  millisecond of perceived delay matters.

RECOMMENDATION  :
  Lambda handlers → Node.js (TypeScript preferred)
  Shared codebase → Same language as Next.js frontend (one team, one language)
  Heavy ML tasks  → Python only if ever needed (separate Lambda function)

  If team is Python-first: stick with Python BUT combine with
  Provisioned Concurrency (Enhancement #2). Do not do neither.

BENEFIT         : Significantly faster cold starts, shared language with frontend team.
IMPLEMENTATION  : Medium effort — rewrite Lambda handlers in Node.js/TypeScript.


--------------------------------------------------------------------------------
ENHANCEMENT #6 — Add DynamoDB TTL for Session Cleanup
--------------------------------------------------------------------------------
ORIGINAL SPEC   : No data lifecycle management mentioned
PROBLEM         : Every chat session — including abandoned/bounced ones — accumulates
                  in DynamoDB forever. At scale:
                    - Table bloats with useless incomplete sessions
                    - DynamoDB storage costs increase
                    - Query performance degrades over time

RECOMMENDATION  :
  On session creation, set TTL field:
    Incomplete sessions : TTL = 30 days (auto-deleted by DynamoDB)
    Completed sessions  : Archive to S3 first, then TTL = 90 days in DynamoDB

  DynamoDB TTL deletes expired records automatically — no cron job needed.
  Completed leads preserved in S3 for full 7-year archival window.

  Session record example:
    {
      "session_id": "abc123",
      "is_complete": false,
      "ttl": 1234567890   <- Unix timestamp, DynamoDB handles deletion
    }

BENEFIT         : Automatic data hygiene. Keeps DynamoDB lean and costs low.
IMPLEMENTATION  : Low effort — add TTL field on session creation.


--------------------------------------------------------------------------------
ENHANCEMENT #7 — Use AWS Secrets Manager for All API Keys
--------------------------------------------------------------------------------
ORIGINAL SPEC   : Anthropic API key mentioned but storage method unspecified
PROBLEM         : API keys stored as Lambda environment variables are visible
                  in PLAIN TEXT in the AWS console to anyone with IAM access.
                  A misconfigured IAM policy = instant credential exposure.

RECOMMENDATION  :
  Store in AWS Secrets Manager:
    - Anthropic / OpenAI API key
    - Admin dashboard API key (from Enhancement #4)
    - Any future third-party service keys

  Lambda fetches secret at cold start → caches in memory for warm requests
  → Never visible as plain text in AWS console

  Cost: $0.40 per secret per month — negligible.

BENEFIT         : Credentials never exposed in console. Rotatable without redeployment.
IMPLEMENTATION  : Low effort — one Secrets Manager call at Lambda startup.


--------------------------------------------------------------------------------
ENHANCEMENT #8 — Add Conversation Drop-off Tracking
--------------------------------------------------------------------------------
ORIGINAL SPEC   : Conversations logged, but no analytics or funnel visibility
PROBLEM         : Without knowing where users abandon the flow, EBAM cannot
                  improve the chatbot. If 60% of users drop off at "ask_budget",
                  that question needs to be reworded — but no one would know.

RECOMMENDATION  :
  Track on every state transition in DynamoDB:
    {
      "state_id"  : "ask_budget",
      "entered"   : 120,   <- sessions that reached this state
      "completed" : 95,    <- sessions that answered and moved on
      "abandoned" : 25     <- sessions that went silent here
    }

  Display in admin dashboard as a simple funnel:
    start         → 200 sessions  (100%)
    advisor_flow  → 160 sessions  (80%)
    ask_budget    → 120 sessions  (60%)  ← drop-off visible here
    ask_name      →  95 sessions  (47%)
    lead_captured →  80 sessions  (40%)

  No third-party analytics tool (Mixpanel, Amplitude) needed.
  All data stays in EBAM's own AWS account.

BENEFIT         : EBAM can optimise flows without developer involvement.
                  Directly improves lead capture rate over time.
IMPLEMENTATION  : Low effort — increment counters on each state transition.


--------------------------------------------------------------------------------
ENHANCEMENT #9 — Widget as Embeddable Script Tag (Not iframe)
--------------------------------------------------------------------------------
ORIGINAL SPEC   : "Website widget deployment flow" — implementation unspecified
PROBLEM         : iframe embeds are severely limited:
                    - Cannot match client website fonts or brand colours
                    - Fixed positioning issues on mobile
                    - Cannot respond to parent page events
                    - Every client site looks different — iframe looks foreign

RECOMMENDATION  :
  Client pastes ONE line into their website:
    <script src="https://widget.ebam.io/loader.js?agent=CLIENT_ID"></script>

  The loader script:
    → Injects chat widget directly into the client's DOM
    → Fetches agent config (colours, name, avatar, flow) using CLIENT_ID
    → Fully isolated from client's existing CSS (Shadow DOM)
    → EBAM updates all client widgets instantly by updating the CDN file
    → No client involvement required for updates

  This is the same pattern used by Intercom, Drift, and HubSpot.
  CLIENT_ID maps to the agent configuration in EBAM's admin dashboard.

BENEFIT         : Clean embed, brand-matched widget, zero-touch client updates.
IMPLEMENTATION  : Medium effort — build widget as standalone bundle + loader script.
